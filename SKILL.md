## 核心理念：慢整理 (Slow Organizing) 与 场景化规划

**"整理看得见的物品，改变看不见的世界。"**
1. **慢整理**：不追求一次性永不复乱，通过"0+4 模式"建立可持续秩序。
2. **MECE 原则**：分类的科学标准 —— "相互独立，完全穷尽"。
3. **动线与频率原则 (Accessibility)**：
   - **黄金位置 (Golden Zone)**：视线与腰部之间，触手可及。存放【高频使用】物品。
   - **收纳逻辑**决定位置。
4. **场景化规划 (Activity-based Planning)**：
   - **核心逻辑**：空间除了区分频率，更要定义其**"主体活动"**。

## 数据操作避坑与分类经验

- **JSON 数据结构陷阱**：遍历 `data.get("items", [])` 而非直接遍历 `data`。
- **批量数据迁移的坑（极易踩坑）**：当编写一次性脚本批量修改本地 JSON 数据（如全量中英文状态转换）时，**必须同时更新被修改项的 `updated_at` 字段**（设为 `datetime.now().isoformat()`）。否则，在执行双向同步脚本（如 `sync_final.py`）时，由于云端记录的更新时间 ≥ 本地的旧时间，双向比对逻辑会误判云端为最新，从而将本地刚修改好的新数据全部反向覆盖回旧版本。
- **“新脚本探索”工作流防御**：在涉及真实飞书数据的双向同步或架构重构时，切忌直接在已有核心脚本中堆砌一次性迁移的 `if/else` 逻辑。**最高效安全的策略**是：
  1. 编写独立的一次性脚本（如 `migrate_xxx.py`）进行批量处理或边界试错。
  2. 跑通并验证云端数据正确同步后，将有效逻辑合并回主干。
  3. 彻底删除过渡脚本并提交，保持 `scripts/` 目录无历史包袱。
- **物品分类边界案例**：
  - **空包装/空盒子**：属于 `🏠 家居与工具 -> 包装与收纳`。
  - **蒸汽眼罩**：属于 `🧼 个人护理 -> 身体洗护`。

### 物品清单关键字段

| 字段名 | 类型 | 说明 |
|---|---|---|
| 分类表 | 关联(link) | 链接到分类表的 record_id |
| 大分类 | 查找引用(lookup) | 从分类表自动填充，**不需要推送** |
| 大类 | 单选(select) | 冗余字段，由查找引用自动覆盖 |
| 子分类 | 文本(text) | 手动推送 |

### 唯一标识与同步强一致性

1. **本地 ID (`id`)**：唯一准则。
2. **唯一入口**：`scripts/sync_final.py`。

### 字段映射配置（field_mapping.json）

**设计目的**：将字段 ID 从代码中抽离，集中维护。飞书端重命名字段或重建表时，**只需修改 JSON 配置，无需改动 Python 代码**。

**配置结构**：
```json
{
  "tables": {
    "items": {
      "table_id": "tbluMVXBpHIJDGyi",
      "fields": {
        "name": {"feishu_name": "物品名称", "feishu_id": "fldILOfvTU", "type": "text"},
        "category_link": {"feishu_name": "分类表", "feishu_id": "fldtoKBC3c", "type": "link"}
      }
    }
  }
}
```

**使用方式**：
- 脚本中通过 `load_field_mapping()` 加载配置
- 通过 `get_field_id(mapping, table_key, field_key)` 获取字段 ID
- 读取阶段：`field_id_list` 建立 `字段 ID → 列索引` 映射
- 推送阶段：JSON key 使用字段 ID（如 `{"fldILOfvTU": "牙膏"}`）

**维护流程**：
1. 飞书重建表 → 用 `lark-cli base +record-list` 获取新字段 ID
2. 更新 `field_mapping.json` 中的 `feishu_id`
3. 同步脚本无需任何改动

### 关联字段必须传 Record ID

1. **核心规则**：飞书多维表格的"关联字段"（如"分类表"链接到 tbl6Ew6fmmhqeeSP），**必须传被关联表的 record_id**，传纯文本无效。
2. **映射字典**：`data/category_mapping.json` 维护了子类名称→`{record_id, major}` 的完整映射，同步时必须从此字典查 record_id。
3. **推送格式**：`{"fldtoKBC3c": "recvhlcRUjNcMt"}`（使用字段 ID），lark-cli 会自动转换为 `[{"id": "recvhlcRUjNcMt"}]`。
4. **本地字段**：items.json 中每个物品的 `category_record_id` 存储对应的分类表 record_id。
5. **深度比对**：`is_content_equal` 必须包含 `category_record_id` vs 云端关联字段的比对，避免遗漏推送。
6. **大类不需要推送**：飞书的"大分类"是查找引用字段，会从关联的分类表自动获取大类，推送时跳过"大类"字段。

### 分类表双向同步

1. **脚本**：`scripts/sync_category_mapping.py`
2. **用法**：
   - `--pull`：从飞书拉取分类到本地（飞书新增的分类同步到本地）
   - `--push`：将本地分类推送到飞书（本地新增的分类同步到飞书）
   - `--sync`：双向同步（推荐日常使用，先 pull 再 push）
3. **维护流程**：
   - 在飞书调整分类后 → `--pull` 或 `--sync`
   - 在本地 `category_mapping.json` 新增分类后 → `--push` 或 `--sync`
4. **禁止**：不再使用手动导出 CSV 的方式更新映射，统一通过脚本同步。

## 同步性能与内容一致性

1. **内容指纹校验**：在向飞书推送更新前，系统必须进行"深度内容比对"。
2. **核心字段范围**：`物品名称`、`位置`、`容器`、`分类表`（关联）、`子分类`、`备注`。
3. **静默跳过策略**：若内容一致，即使本地时间较新，也应**跳过** API 请求，保护飞书"更新时间"真实性。

## 历史数据迁移与实物对照逻辑

### 事实驱动原则

1. **禁脑补列表**：严禁通过分类联想自行添加源文件中不存在的物品。
2. **子类导向迁移**：以新系统子类为锚点，去历史库中打捞匹配项。
3. **实物找寻与安置**：迁移是找回实物并赋予新家（位置/容器）的过程。
4. **位置与容器联动**：移动某个容器时，AI 自动批量更新其内部所有物品的位置。

## 空间表设计（自引用无限嵌套）

### 设计原理

采用**统一树形模型**：一张空间表，通过自引用关联字段 `所属空间`（link → 本表自身）实现无限层级嵌套。

```
次卧 (room, parent: null)
├── 桌子 (surface, parent: 次卧)
│   └── 箱子 (container, parent: 桌子)
│       ├── 袋子A (container, parent: 箱子)
│       └── 盒子B (container, parent: 箱子)
├── 衣柜 (furniture, parent: 次卧)
│   ├── 挂杆 (sub_area, parent: 衣柜)
│   └── 抽屉 (sub_area, parent: 衣柜)
```

### 空间类型枚举 (type)

- `room` - 房间（顶层，无 parent）
- `area` - 区域（如床区、吧台、暂存区）
- `surface` - 表面/台面（如桌面、操作台）
- `furniture` - 家具（如衣柜、鞋柜）
- `container` - 容器（如箱子、袋子、盒子、收纳袋）
- `sub_area` - 家具内部分区（如抽屉、隔层、挂杆）
- `appliance` - 电器（如冰箱，内部可挂子容器）
- `wall` - 墙面
- `cabinet` - 柜子
- `organizer` - 收纳件（如刀架）
- `wall_hook` - 墙面挂钩

### 飞书空间表字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| 空间名称 | 文本 | 显示名称 |
| 所属空间 | 关联(本表自身) | 自引用，实现无限嵌套，传 record_id |
| 空间类型 | 单选 | 上述 type 枚举 |
| 使用频率 | 单选 | high/medium/low |
| 盘点状态 | 单选 | pending/active/inventory_completed/pending_reorganize |

### 关键技术发现

1. **飞书多维表格支持自引用关联**：创建关联字段时 `link_table` 设为本表 ID 即可。
   - 创建命令：`lark-cli base +field-create --json '{\"field_name\":\"所属空间\",\"type\":\"link\",\"link_table\":\"tbl2cA30WNdkjsLS\"}'`
2. **嵌套推送格式**：`{\"所属空间\": \"parent_record_id\"}`，传父节点的 record_id。
3. **已验证4层嵌套**：次卧→桌子→箱子→袋子，均成功创建关联。
4. **旧字段处理**：原表的 `空间`(单选) 和 `容器`(单选) 字段需评估是否保留或替换。

### 飞书字段类型辨析：查找 vs 查找引用

创建层级路径显示时，飞书有两种相似的字段类型：

| 类型 | 图标 | 特点 | 适用场景 |
|------|------|------|----------|
| **查找** | 🔍 放大镜 | 需手动设置查询条件，从其他表按条件搜索 | 复杂条件查询 |
| **查找引用** | 🔗 双表箭头 | 从已有关联字段自动获取数据，无需条件 | 关联字段的值传递 |

**创建查找引用的正确步骤**：
1. 添加字段 → 类型选择「查找引用」（不是「查找」）
2. 选择关联字段：如 `所属空间`
3. 选择引用字段：如 `空间名称`
4. 无需设置查询条件，自动从关联记录获取值

**链式查找引用方案**（用于显示完整路径）：
1. 创建 `父级名称`：查找引用 ← 从 `所属空间` 获取 `空间名称`
2. 创建 `祖父级名称`：查找引用 ← 从 `所属空间` 获取 `父级名称`
3. 创建公式字段拼接路径

> ⚠️ 注意：飞书查找引用可能不支持链式引用（lookup 的 lookup），需实测验证。

### 空间表双向同步 (最新版)

1. **脚本**：`scripts/sync_spaces_bidirectional.py`
2. **映射文件**：`data/space_tree.json`（树形统一模型，包含节点数组及层级关联关系）
3. **映射结构示例**：
   ```json
   {
     "nodes": [
       {
         "record_id": "recvhqPgaJU2PG",
         "name": "杂物间柜子",
         "type": "柜子",
         "id": "杂物间_收纳架",
         "parent_id": "virtual_过道/杂物间"
       }
     ]
   }
   ```
4. **功能特性**：
   - **自动双向比对**：逐字段比对节点名、类型、备注、使用频率、盘点状态等。
   - **云端拉取更新**：飞书数据修改自动覆盖本地。
   - **本地推送新建**：本地新增节点自动推送飞书并回写 `record_id`。
   - **云端发现新增**：飞书新增空间自动发现并拉取到本地，分配本地 ID。
   - **树形关系维护**：无论哪端修改 `parent_id`，均可自动梳理和同步层级引用。
   - **自动处理枚举格式**：自适应飞书单选/多选字段所需的数组封包逻辑。

### 字段映射双向同步

1. **脚本**：`scripts/sync_field_mapping.py`
2. **功能特性**：
   - 自动检测飞书列名（`feishu_name`）的文字变更。
   - 依赖不随重命名而改变的 `feishu_id`。
   - 随时保持本地 `field_mapping.json` 中名字的最新状态和可读性，防漂移。
